   FailedFallback}`. `FallbackSpec { path, format }`,
   `FallbackFormat::{Nexus, Rkyv}`.

## 2 · cli-msg policies

The skeleton commits to the *wire shape*. Policies (timing,
file paths, ID strategy, state machine) live here — easier to
tune as implementation lands.

### 2.1 · Heartbeat

- **Interval default**: 5 seconds. nexus-cli sends the first
  heartbeat 5s after a `Send` if no reply has arrived; subsequent
  intervals exponentially back off (5 → 10 → 20 → 60s, capped).
- **Liveness threshold**: two consecutive heartbeats without
  reply ⇒ assume socket is dead.
- **On dead socket**: nexus-cli attempts one reconnect; if that
  fails, falls back to polling the fallback file (if the client
  supplied one). If neither works, exit with `daemon
  unreachable, fallback unavailable`.
- **Heartbeat for unknown `cli_request_id`** (nexusd has no
  record — process restart, connection reset): reply
  `CliReply::Failed { error: "request id not found; socket may
  have been reset" }`. Client interprets: original lost; use
  Resume if fallback exists, or retry from scratch.
- **nexusd does not forward heartbeats to criomed.** Heartbeat
  state is local; nexusd answers from its own in-memory
  `RequestState`. Criomed never sees these probes.

### 2.2 · Fallback file

- **Path**: client-supplied (`FallbackSpec::path`). No nexusd
  default — the client decides where state goes; nexusd never
  surprises with file writes outside the supplied path.
- **Atomic write**: `${path}.in-progress` is written, then
  `rename(temp, path)`. Polling clients never observe a
  half-written file.
- **Mode**: 0600 (owner-only). Replies may carry sensitive
  diagnostics or intermediate state.
- **Format default at rung 1**: `FallbackFormat::Nexus` (human-
  readable text). `Rkyv` is available for tools that prefer the
  canonical binary form. Client picks per request.
- **Cleanup**: on successful `Resume`, nexusd reads the file
  and deletes it. Orphaned files (client never resumed) are
  the client's TTL/cleanup responsibility (e.g., `find
  ${fallback_dir} -mtime +7 -delete`).
- **Concurrent resumes**: first `Resume` to acquire the file
  wins (read + delete is atomic); the second gets
  `ResumeNotReady`.

### 2.3 · `CliRequestId` strategy

- **Format**: UUID v7 (time-ordered, globally unique). 16
  bytes. Generated by nexus-cli; opaque to nexusd; never seen
  by criomed.
- **Why UUID v7**: collision-free across processes (clients
  sharing a fallback dir don't step on each other), time-
  ordered (logging stays sortable), unguessable (logs in bug
  reports don't leak structure).
- The `CliRequestId(u64)` newtype in the skeleton is wrong-
  sized for UUID. Fix at implementation time:
  `CliRequestId([u8; 16])` or `pub struct CliRequestId(pub
  Uuid)`.

### 2.4 · nexusd state machine (per in-flight request)

| State | Entry | Transitions | Persistence |
|---|---|---|---|
| Idle | connected; no Send outstanding | `Send` → Parsing | memory |
| Parsing | `Send` received | parse ok → BuildingMsg; parse fail → Failed | memory |
| BuildingMsg | AST ready | build ok → AwaitingCriomed; build fail → Failed | memory |
| AwaitingCriomed | criome-msg sent to criomed; reply pending | criomed reply → Serialising; client socket drop → WriteFallback | memory + fallback path (if supplied) |
| Serialising | criomed reply received; encoding nexus text | encode ok → Done; encode fail → Failed | memory |
| WriteFallback | client socket dropped mid-flight | atomic write ok → FallbackWritten; write fail → FailedFallback | **persistent** (file on disk) |
| Done | reply ready or fallback written | reply sent → Idle; or cleanup → Idle | memory |
| Failed | parse / build / criomed-transport error | error sent → Idle | memory (no fallback) |
| FailedFallback | fallback write failed | error sent → Idle | memory |

Persistence boundary: only `FallbackWritten` survives nexusd
restart. Heartbeats for in-memory state during `Awaiting…`
return `Working { stage: AwaitingCriomed }`.

### 2.5 · Cancel semantics

- Cancel for already-replied request → `Cancelled` (no-op
  acknowledgement).
- Cancel for `AwaitingCriomed` → nexusd discards the eventual
  criomed reply, replies `Cancelled` immediately. Criomed is
  not asked to cancel (criome-msg v0.0.1 has no cancel verb;
  added if real demand surfaces).
- Cancel for unknown request_id → `Failed { error: "request id
  not found" }`.

### 2.6 · Connection model

- Single socket path: `${XDG_RUNTIME_DIR:-/tmp}/nexus-cli.sock`
  with optional symlink at `~/.nexus/cli.sock`.
- One nexus-cli invocation = one connection. Multiple
  in-flight requests on one connection are allowed but
  unusual.
- nexusd-side correlation: `HashMap<(connection_id,
  CliRequestId), RequestState>`. Per-connection isolation is
  the OS's job.

## 3 · Wire format — the frame schema is the framing

Both cli-msg and criome-msg wires carry a stream of frame
archives — `Archived<CliFrame>` and `Archived<Frame>`
respectively. Both parties know the schema, so framing is
intrinsic to it: the universal handshake message is the frame
type itself. No "raw bytes outside rkyv" layer; the schema
encodes whatever structure delimiters the implementation
needs.

reports/070 §6 updated to match.

### 4.3 · Genesis self-loop resolution

At first boot, criomed needs to dispatch `genesis.nexus`. Two
candidate architectures:

(a) criomed reads the file, parses it via `nota-serde-core`
in-process, builds criome-msg envelopes, writes records to sema
directly. **Wrong** — bypasses nexusd; reintroduces the "two
journeys" hand-wave [reports/065 §2](repos/mentci-next/reports/065-criome-schema-design.md)
catalogued as wrong-framing. Even though the resulting bytes
match what nexusd would produce, the path violates the rung-by-
rung rule.

(b) criomed reads the file (`include_str!`), opens a UDS
client to nexusd, sends each `(Assert ...)` as a
`CliRequest::Send`. nexusd parses, builds criome-msg, sends
*back* to criomed via the criome-msg socket. criomed validates
and writes. **Correct** — the genesis stream is a stream like
any other; nexusd's role is honoured; multi-process from day
one holds.

The "self-loop" objection is conceptual not real — it's a
Unix-socket loop, not a control-flow loop. Tokio handles it
trivially.

This makes the "built-in Rust types" usage in criomed's
schema-check step a *fallback during the genesis stream*
(when no in-sema KindDecl exists yet for the kind being
asserted), not a primary input path. Mechanism stays
explainable.

## 5 · Implement-vs-design split

**Skeleton-as-design dominates this phase.** No more pure
prose reports until the skeleton compiles end-to-end.

- Real implementation is appropriate for tightly-bounded leaf
  functions whose shape is unambiguous: `CliRequestId::fresh()`
  (uuid v7), `CliFrame::encode`/`decode` (rkyv API call), redb
  table opens.
- Real implementation is *premature* for criome-msg's verb
  handlers, criomed's validator pipeline steps, nexusd's cli-
  msg listener — interfaces are still adjusting; locking
  rkyv-derive-checked types catches drift faster than locking
  bodies.
- After the cargo-check-workspace-passes milestone (Step 8),
  the next phase fills bodies along the narrowest path to the
  Stage A success criterion.

Mix recommendation: **80% skeleton-as-design + 15% trivially-
bounded real bodies + 5% genesis.nexus text + 0% new prose
reports**.

## 7 · Three new questions for Li

### Q3 · `Slot` / `Revision` / `Blake3Hash` newtype location

These newtypes appear in `criome-msg` (wire), `criome-schema`
(record fields), `sema` (table keys). Three candidates:

(i) Duplicate per crate (each derives rkyv separately).
(ii) Define in `criome-schema`; re-export from `criome-msg`
and `sema`.
(iii) Define in a tiny `criome-types` crate that all three
depend on.

Lean: (iii) for cycle-safety and dependency-graph cleanliness.
Confirm.

### Q4 · `CliRequestId` size — keep `u64` or switch to UUID v7?

The skeleton currently has `pub struct CliRequestId(pub u64)`.
Policy recommendation in §2.3 is UUID v7 (16 bytes). Confirm
the size change at implementation time, or stick with u64 +
monotonic counter per process.

### Q5 · Cancel-criomed verb — add to criome-msg or skip?

cli-msg has a `Cancel` verb; nexusd handles it locally by
discarding the criomed reply. If real workflows need to abort
expensive criomed operations (long-running queries, complex
cascade evaluations), criome-msg would need a parallel cancel
verb. Add now (for future-proofing) or wait until demand
surfaces?
